# 現在的篩選條件是否太嚴格？

**是。以「探索 NL pathway」這個目的來說，目前流程非常嚴格，偏向高可信度、低敏感度。**

目前 66 個 NL-up genes 要同時通過：

```
每一個資料集內 adjusted P < 0.05   
每一個資料集內 |log2FC| > 1   
RNA-seq 還要 baseMean ≥ 10   
四個資料集都要出現   
四個資料集都要是 NL 高於 LS   
HGNC mapping 後不能有衝突   
ORA pathway 至少命中 3 genes   
最後還要通過 BH adjusted P ≤ 0.05   
```

這些條件疊加後，得到 66 genes 並不奇怪。

多資料集研究常出現「不同研究的 individual DEGs 重疊不高，但受影響的 pathways 比較一致」的情況。Bioinformatics 的 pathway meta-analysis 研究也指出，將資訊整合到 pathway 層級通常比只取各研究 DEG 的硬性交集更有檢定力。

因此問題不是：

```
要不要為了多找一些結果，任意放寬？
```

而是：

```
目前的 4/4 strict DEG intersection 是否只適合找最核心 genes，卻不適合當成唯一的 NL pathway discovery 方法？
```

我的判斷是：是，目前它太嚴格，不應成為唯一分析。

# Q:好，那我先做完我現在這版的ORA+GSEA後，看看結果如何，並幫我判斷，是否需要再做3/4 same-pattern ORA跨資料集敏感度分析，你覺得如何?

# A:我覺得這個順序很合理，而且比現在立刻加做 3/4 ORA 更好。

先完成目前已定義的：

1. 4/4 strict ORA
2. 四資料集 full-ranked GSEA
3. Exclude／Split 比較
4. 四資料集 GSEA 結果整合
5. ORA–GSEA 方法學互證

之後再判斷 3/4 same-pattern ORA 是否有必要。這樣不會讓分析流程一開始就變得太散，也能先看完整排名的 GSEA 能不能補足目前 NL-up ORA 訊號較少的問題。

## 但判斷標準最好現在先固定

不要等看到結果後，單純因為「pathways 太少」才決定加做。建議事先規定：

### 建議加做 3/4 same-pattern ORA 的情況

* 4/4 strict NL-up ORA 仍只有很少 pathways，或主要集中在單一 pathway family。
* GSEA 發現多條有生物意義的 NL-up pathways，但多數只在 3/4 資料集中顯著，而不是四個都顯著。
* 某一個資料集的訊號明顯較弱，但另外三個資料集結果一致，懷疑是樣本數、平台或統計檢定力造成。
* ORA 與 GSEA 顯示相近的生物主題，但 strict 4/4 gene overlap 沒有足夠 genes 支撐 ORA。
* Exclude 與 Split 的主要結論大致穩定，表示結果不是由 multi-symbol probe 處理方式造成。

### 可以不做的情況

* 新版 GSEA 已經找到足夠且清楚的 4/4 NL-associated pathways。
* ORA 與 GSEA 已能互相支持主要生物機制。
* 3/4 結果沒有形成一致的生物主題，只是增加很多零散 pathways。
* GSEA 本身也沒有顯示可信的 NL-up 訊號，此時放寬到 3/4 ORA 未必能真正解決問題。

## 3/4 same-pattern 建議怎麼定義

最穩健的規則不是「任意三個資料集顯著就算」，而是：

> 至少 3/4 資料集通過原本 DEG 門檻，且三個資料集皆呈現相同的 LS／NL 表現差異；剩下的一個資料集不得出現顯著的相反結果，最好連 logFC 正負也相同。

例如 NL-up gene：

* 至少三個資料集通過 DEG 門檻，且都是 NL 高於 LS。
* 第四個資料集可以未達顯著，但不能顯著呈現 LS 高於 NL。
* 仍使用目前已建立的 common universe。
* Exclude 仍為主版本，Split 仍為敏感度版本。
* 3/4 ORA 只能定位為補充性的跨資料集敏感度分析，不能取代 4/4 strict 主分析。

## 最終建議

你現在先不要重開新的 DEG 門檻，也不要先做 3/4 ORA。

目前最好的路線是：

> 先完成新版 GSEA → 看 NL-up pathways 在四個資料集中的支持情況 → 再依預先固定的判斷標準，決定是否加入 3/4 same-pattern ORA。

以你現在 strict NL-up ORA 只有一個主要 Reactome 訊號家族的情況，我認為 **3/4 ORA 很可能最後會有補充價值**；但是否真的需要，應等新版 GSEA 結果出來後再正式判定。
