# 一、四個資料集做 4/4 與 Strict 3/4 consensus：合理，但不是正式 meta-analysis

你目前的規則：

* 4/4：四個資料集皆顯著且方向一致
* Strict 3/4：三個資料集顯著，第四個方向相同但未顯著
* 排除第四個方向相反
* 排除未測試的 3/4-U，或只列補充

這是一個**保守而清楚的跨資料集重現性篩選法**。尤其 Strict 3/4 沒有要求第四個資料集一定顯著，但要求 NES 方向一致，可以避免樣本數或平台差異造成「方向一樣但差一點沒顯著」就被完全排除。

不過，這種方法本質上仍屬於：

> **significance-based consensus／vote-counting filter**

而不是正式的統計 meta-analysis。Bioinformatics 的 pathway meta-analysis 研究指出，單純取各研究交集可能過度保守，而在 pathway level 結合各研究的統計證據，通常能獲得更高統計力。

### 你的正確定位

目前可稱為：

> **Cross-dataset consensus pathway analysis**

不要直接稱為：

> Pathway meta-analysis

論文中可寫：

> Pathways were defined as cross-dataset consensus signals when they were significant in at least three datasets and showed concordant NES directions across all four datasets.

這樣完全合理。

### 之後可以加強，但不是現在非做不可

後續若時間足夠，可以加入：

* pathway-level P-value combination
* NES effect-size meta-analysis
* leave-one-dataset-out stability analysis

但對目前的 short paper 而言，你的 4/4＋Strict 3/4 可先作為清楚、保守的 discovery framework。

# 二、Microarray 分 Exclude 與 Split：問題值得處理，但兩者地位不能相同

## Exclude 當正式主分析：合理，而且文獻支持最強

Microarray probe 到 gene 的 many-to-many mapping 是公認的重要問題。經典 Nucleic Acids Research 研究顯示，更新 probe definition 後，不同分析得到的 DEG 清單可能產生約 30%–50%差異，而且會進一步影響 GSEA 與 Fisher’s exact test 等功能分析。

PLoS Medicine 的 microarray meta-analysis 指南也把：

> 解決 probe 與 gene 的 many-to-many mapping

列為正式流程中的關鍵步驟之一。([PLOS][2])

在實際 Bioinformatics 研究中，對同時 hybridize 到多個不同 genes 的 ambiguous probes，常見的保守做法就是直接排除。

所以：

> **Exclude 作為正式主分析是合理且最容易辯護的。**

## Split 當 sensitivity analysis：可以，但不應作為平行主分析

Split 是將：

> GENE_A /// GENE_B

拆成兩筆，並把同一個 probe 的統計量分配給兩個 gene。

它可以回答一個有價值的問題：

> 如果不丟掉 ambiguous probes，而是盡量保留其可能對應基因，主要 pathway 結果會不會改變？

因此 Split 適合作為：

> **annotation sensitivity analysis**

但它有明顯限制：同一筆 probe 證據被複製到多個 genes，可能增加某些 gene sets 的命中數或改變 ranked list。因此不建議把 Split 與 Exclude 當成兩套同等地位的正式分析，更不能把兩者當成第五、第六個獨立資料集。

### 最合理設計

* **Primary analysis：Exclude**
* **Sensitivity analysis：Split**
* 比較：

  * gene ranking correlation
  * NES correlation
  * 顯著 pathway overlap
  * NES direction concordance
  * 4/4／Strict 3/4 分類是否變動
  * 主要功能結論是否穩定

如果結果一致，Split 的價值很高；如果結果不一致，就標示哪些 pathways 對 annotation strategy 敏感。

### 更理想但不一定做得到

若原始平台及 raw data 支援，更新 probe annotation 或使用重新定義、唯一對應的 probe sets，通常比直接 Split 更乾淨。NAR 的研究正是強調更新 probe definitions 對結果的重要性。

# 三、同時做 ORA 與 GSEA：合理，但不要稱為「雙重獨立驗證」

## 為什麼合理？

ORA 與 GSEA 回答的問題不同：

* **ORA**：通過 DEG 門檻的 genes 是否在某 gene set 中過度出現？
* **GSEA**：整條完整 ranked list 中，某 gene set 是否整體集中於排序的一端？

ORA 依賴 DEG cutoff；GSEA 則利用完整基因排序，能捕捉單一 gene 變化不強、但整組 genes 同方向移動的訊號。原始 GSEA 論文就是為了解決只看單一 DEG 容易遺漏協同變化的問題。([PMC][3])

Genome Biology 的 pathway analysis 綜合評估也將 ORA 歸為使用 DEG list 的第一代方法，而 gene-set ranking methods 則保留更多連續統計資訊。([Springer][4])

因此同時做兩者是合理的，因為它能辨認：

* ORA 與 GSEA 皆支持
* 僅 GSEA 支持：可能是整體小幅協同變化
* 僅 ORA 支持：可能由少數強 DEG 驅動
* 方向或證據層級不一致

## 但「雙重驗證」用詞太強

ORA 與 GSEA：

* 使用同一批表現資料
* 通常使用相同或高度重疊的 gene sets
* 統計結果並不互相獨立

所以兩者一致只能表示：

> 結果對 DEG cutoff 與 enrichment method 的選擇具有一定穩健性。

不能表示：

> 已經被兩個獨立實驗驗證。

建議用詞：

* **complementary pathway analyses**
* **methodological triangulation**
* **ORA–GSEA concordance analysis**

而不要寫：

* independent dual validation

另外，ORA 的 universe 必須使用該資料集中**實際通過前處理且可被檢驗的 genes**，不能直接使用全人類基因組；否則 RNA-seq 與 microarray 的可檢測範圍不同，會造成 enrichment bias。

# 四、使用 Hallmark、Reactome、GO BP：合理，而且組合很好

但應把「三個 pathway databases」改成：

> **三個 gene-set collections**

因為：

* Hallmark：精簡的生物狀態與程序 signatures
* Reactome：較具機制性、人工整理的 molecular pathways
* GO BP：階層式 biological-process ontology，嚴格來說不完全等同 pathway database

Nature Communications 的整合 enrichment 方法研究指出，GO biological processes 與 Reactome molecular pathways 是最常使用的兩類功能 gene sets。([Nature][5])

Hallmark 則是專門將大量重複、異質的 MSigDB founder sets 整理成較精簡且表現一致的 50 組 signatures，用意本來就是降低雜訊與冗餘。([PMC][6])

### 這三者剛好有互補性

| Collection | 適合用途                 |
| ---------- | -------------------- |
| Hallmark   | 看主要生物軸線，最容易寫簡報與摘要    |
| Reactome   | 看較細的分子機制與 pathway 模組 |
| GO BP      | 看廣泛的生物過程、發育與結構功能     |

因此使用三個 collections **不是過度分析**。真正需要注意的是：

1. 每個 collection 分開做 BH correction。
2. 報告確切 MSigDB 版本。
3. 不要把三個資料庫中高度重複的結果視為三次獨立證據。
4. 不要把三個 collection 的 pathway 數量直接加總後稱為獨立生物機制數量。

Hallmark 可以作為主要敘事骨架，Reactome 與 GO BP 用來增加細節與支持。

# 五、Reactome 與 GO BP 去冗餘及功能分群：非常合理，甚至是必要步驟

你現在得到數百條 Reactome 與 GO BP terms，如果全部逐條解讀，會有兩個問題：

* 許多 terms 共享大量 genes。
* GO 本身具有 parent–child 階層，相似名稱可能只是同一生物概念的不同層次。

Enrichment Map 的方法就是將 gene sets 當成網路節點，再依 Jaccard／overlap coefficient 連結重疊 terms，將大量 enrichment results 整理成功能群組。([PLOS][7])

REVIGO 也是因為 GO enrichment lists 通常過長且高度冗餘，而使用 semantic similarity 找出代表性 GO terms。([PLOS][8])

更新的 simplifyEnrichment 則可直接對功能 terms 的 semantic-similarity matrix 進行 clustering。([OUP Academic][9])

## 建議不要用同一種方式處理兩個 collection

### GO BP

優先使用：

* GO semantic similarity
* simplifyEnrichment
* REVIGO

因為 GO 有正式的 ontology hierarchy。

### Reactome

優先使用：

* gene overlap
* Jaccard coefficient
* overlap coefficient
* EnrichmentMap
* Reactome hierarchy

因為 Reactome 的 terms 是 pathway membership 與上下層機制關係，不完全適合只用 GO semantic similarity。

## 去冗餘不代表刪除原始結果

正確做法是：

* 完整結果保留在 Supplementary Table。
* 主文顯示功能 clusters。
* 每個 cluster 選 1–3 條代表 pathways。
* 保留 cluster 中所有原始 pathways 的對照表。

推薦分群可包含：

* 免疫與發炎
* 細胞週期與增殖
* 組織結構與 ECM
* 屏障與細胞連結
* 發育與分化
* 脂質與代謝穩態
* 神經／間質／收縮相關訊號

這一步是必要的，不是多做無用工作。

# 六、檢查代表性 pathways 的 leading-edge genes：合理，但應放在去冗餘之後

原始 GSEA 將 leading-edge subset 定義為：在 enrichment running score 達到最大偏移前，最主要貢獻 enrichment signal 的那部分 gene-set members。([PMC][3])

所以檢查 leading-edge genes 能回答：

* 同一功能 cluster 是否由同一批 genes 推動？
* 哪些 genes 在多個 pathways 重複出現？
* 哪些 genes 在四個資料集反覆出現？
* pathway 名稱看似不同，是否其實由相同核心 genes 驅動？

原始 GSEA 研究也曾比較多條相關 pathways 的 leading-edge overlap，以找出共同驅動的 MAPK 元件，說明這種分析本身是正規且有意義的。([cs.tufts.edu][10])

## 你現在的分析順序應該是

1. 先完成 ORA–GSEA 整合。
2. Reactome／GO BP 去冗餘及功能分群。
3. 每個 cluster 選代表性 pathways。
4. 再分析代表性 pathways 的 leading-edge genes。

不應一開始就檢查數百條 pathways 的所有 genes，否則會產生非常大量且難以解讀的 gene lists。

## 建議對每條代表 pathway 記錄

* 完整 gene set
* 各資料集 leading-edge genes
* 4 個資料集出現頻率
* leading-edge union
* leading-edge intersection
* 每個 gene 的排名與方向
* 是否同時出現在其他代表 pathways
* 是否可能反映特定細胞比例

### Strict 3/4 的正確處理

* 「共同 leading edge」先以三個顯著 supporting datasets 計算。
* 第四個未顯著但同方向資料集另外檢查。
* 不要把第四個資料集硬併入正式 intersection。
* 但可以報告第四個資料集中是否也出現相同核心 genes。

## 最重要限制

Leading-edge genes 是：

> 本次資料集、本次比較、本次 gene ranking 下，主要推動 enrichment 的 genes。

它們不是：

* 已證明的 causal genes
* PSO-NL 特異 biomarkers
* 可以直接投入分類器的最終 features

所以這一步適合做**機制解釋與候選優先排序**，不算獨立驗證。

# 最重要的整體風險：分析分支太多，而不是方法不合理

你的每一項方法單獨看都有合理依據；真正可能被審稿人質疑的是：

* Exclude 與 Split
* ORA 與 GSEA
* Hallmark、Reactome、GO BP
* 4/4、Strict 3/4、3/4-U
* 去冗餘前後
* leading-edge intersection／union

若全部都當成同等地位的正式分析，會形成很大的 researcher degrees of freedom。

## 建議預先固定分析層級

### Primary analysis

* Microarray：Exclude
* GSEA：完整 ranked list
* 4/4＋Strict 3/4，同方向
* Hallmark：主要敘事

### Method-concordance analysis

* ORA vs GSEA
* 皆支持／僅 GSEA／僅 ORA／不一致

### Sensitivity analysis

* Microarray Split
* 3/4 ORA
* leave-one-dataset-out
* threshold variations

### Interpretation analysis

* Reactome／GO BP 去冗餘
* 功能分群
* 代表性 leading-edge genes

### Downstream clinical analysis

* PSO-NL vs AD-NL
* PSO-NL vs Healthy
* AD-NL vs Healthy
